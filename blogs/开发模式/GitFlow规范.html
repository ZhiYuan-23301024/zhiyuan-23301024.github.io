    GitFlow概览

    ZhiYuan 8/10/2025 git

    GitFlow规范

    今天在B站上刷到了一条讲解gitflow的视频，感觉讲的非常好且通俗易懂，正好本人对规范这件事情比较不怎么了解，便做个总结笔记，视频链接十分钟学会正确的github工作流，和开源作者们使用同一套流程_哔哩哔哩_bilibili (opens new window)，这个过程我会用草稿纸和作业的关系尽可能讲清楚这个工作流逻辑

    故事概要

    想象一下，老师给你和你的小伙伴们布置了一个小组作业，每个人写其中一部分。于是小组组长手里拿到的是作业本体（远程master分支），即最后需要直接交付老师的作业，组长把它挂在黑板上，让所有人都能看到这份最终作业的进展。组员则在黑板上贴上自己的解答分工部分（远程feature分支）。组长会阅读组员贴在黑板上的这些解答，然后决定是否将这些分工解答的部分写在最终的作业里（合并远程master和远程feature分支）。而组员自己的座位离黑板太远了，于是组员手抄了一份作业到自己的作业本上（拉取远程master分支到本地master分支），免得来来回回跑看作业长什么样。那么写作业的时候肯定是不能直接在作业本上写的对吧，于是组员会自己在座位上拿一本草稿本（本地feature分支）写自己的解题步骤，在草稿本上随便瞎搞都不会影响到作业内容的原始状态。到这就是我们的故事背景。

    拉取远程master到本地master

    在开始开发自己负责的需求之前，拉取远程master分支上的内容到本地master

    #克隆远程master到本地mster
git clone origin master
#若克隆到本地以后本地分支不为master，则将其重命名为master
git branch -m master
#可使用git branch查看分支列表，带*号的为当前分支
git branch

    创建本地feature分支

    本地master分支是和远程master对应的。就像不能轻易直接对远程主分支进行修改一样，我们也不能轻易地直接在master上做自己的功能开发，而是应该从本地master分出一个自己的feature分支，而本地master仅作为提交使用。可以理解为master分支是我们领到的作业，而新建一个feature的目的是打开草稿纸，在草稿纸上写完了才能抄到作业上。

    #从本地master分出一个feature分支 feature分支保留了master分支的所有记录
git checkout -b my-feature
#该过程相当于两条git命令
git branch my-feature #my-feature会继承master的提交
git checkout my-feature #切换当前分支为my-feature

    在本地feature做提交

    当我们在本地feature改完代码之后，就可以在本地feature提交自己的commit了。

    git add <change-file>
git commit -m <change-log>
    1
    2

    将本地feature推送到远程feature上

    当我们在my-feature分支上完成了我们的功能开发以后，我们就可以将它合并到远程my-feature分支上了，即把自己桌上的草稿纸的结果抄到黑板上自己的解答框里，这个时候可以看看作业本体和自己一开始抄下来的比有没有被更改过，若被更改过则进行接下来的操作，若没有被更改过，则可以叫组长过来验收自己的分工了。

    git push origin my-feature
    1

    再次拉取远程master到本地master

    由于项目开发是多线程的，因此你第一次拉取的主分支，经过你开发的这段时间可能已经迭代了，因此你需要把远程最新主分支拉取到本地master，为下一步检查做准备。即看看现在作业整体进度，然后把作业现在进展到哪了，把自己桌上的作业更新一下，再检查自己的解答和最新的作业解答是不是兼容。

    git checkout master #切换到本地master分支
git pull origin master #更新本地master为远程master最新状态
    1
    2

    当然你可能会说，啊老师老师那这个过程中作业还是会继续更新啊，那咋办呢，这个之后会说，别急。

    将本地feature变基到本地master上

    由于本地master此时是最新开发版本，领先了本地feature，因此我们需要将本地feature的提交通过变基的方式与本地master合并，相当于我们的提交是基于最新版本的远程master做的更改。这个过程可能出现rebase conflict，那么就需要手动选择自己想要哪一行代码了。这个过程相当于把自己草稿纸上的内容写到自己抄下来的最新的作业上。

    git rebase master
    1

    将本地feature强制推送到远程feature上

    这个时候我们完成了本地feature和最新master的最新合并测试，确信是可以交上去和远程主分支合并了，因此我们更新远程feature分支为我们本地改好的可以提交的feature分支。即将我们写好的作业（我们负责的那部分）写在黑板上。

    #强制更新远程feature为本地feature
git push -f origin my-feature
    1
    2

    提交pull request

    直接在github仓库里点击new pull request向管理员发起合并请求，相当于告诉组长你的作业分工写完了，可以抄到最后的作业上了。

    组长审核后合并

    仓库管理员审核你的feature和远程master分支的，然后就可以点击仓库里的merge合并你的请求了。恭喜，你完成了一你的作业分工

    结语

    对于这个过程中不详细的部分，比如管理员审核的时候需要审核哪些点，几种合并方式有哪些具体的不同等等，之后的文章里再一一说明
